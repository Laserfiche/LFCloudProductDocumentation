End Events

Every process flow in your process must end with an end event. A generic end event and an error end event terminate only the process flows they are attached to. A terminate end event terminates the entire process, including any sub-processes and process flows, even if these do not end in terminate end events.

Generic End Event

End events mark the end of a process. Once a process flow reaches an end event, the process flow ends. However, if there are other process flows still in progress, those will continue until they also reach an end event.

Example: Within a company, you can use a form to submit an information request from the company's Enterprise Resource Planning (ERP) system. When the form is submitted, it is routed both to a manager for review, and to a Workflow service task which attempts to retrieve the ERP information. If the format of the request is invalid (e.g. it may ask for information that does not exist), the service task routes the process flow to a terminate end event. This means that no matter what happens in other process flows (whether the manager approves or rejects the request), all process flows are terminated. This makes sense because there is no point in asking the manager to approve a request for non-existent information. The other process flows in the process end in generic end events. This means, for example, that when a manager approves or rejects the request, the Workflow service task that retrieves ERP information will not be interrupted.
